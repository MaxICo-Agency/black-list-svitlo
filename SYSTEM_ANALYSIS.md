# Black List Світло парк: системний аналіз

## Що вже зроблено

- Статичний сайт на GitHub Pages.
- Пошук майстра за телефоном.
- Профіль майстра `profile.html?id=...` з телефонами, категоріями, фото робіт і відгуками.
- Дані читаються з Google Sheets CSV листа `Phones`.
- На головній є категорії, рекомендовані майстри та останні записи зі скаргами.
- Локальна резервна копія даних лежить у `data/phones.csv`.

## Поточне обмеження

Перший MVP мав одну строку = один майстер. Це швидко запустило пошук, але така модель погано росте:

- у одного майстра може бути багато телефонів;
- у одного майстра може бути багато категорій;
- у одного майстра може бути багато відгуків;
- фото робіт краще зберігати окремо, а не змішувати з основним рядком.

## Рекомендована модель

На наступному кроці краще зробити кілька листів:

- `Masters`: базова картка майстра або бригади.
- `Phones`: багато номерів для одного `master_id`.
- `Categories`: довідник категорій.
- `MasterCategories`: зв'язок майстра з кількома категоріями.
- `Reviews`: кожен відгук окремим рядком.
- `Photos`: фото робіт або скарг з Google Drive URL.

Так сайт зможе показувати один профіль, багато номерів, багато категорій, історію відгуків і галерею робіт.

## Фото

Для публічного статичного сайту фото з Google Drive мають бути доступні за публічним URL.
Якщо фото і таблиці треба тримати приватними, потрібен серверний endpoint, який читає Google Sheets/Drive через service account і віддає сайту тільки дозволені дані.

## Telegram

Статичний сайт не може надійно й безпечно підтягувати фото/ім'я людини з Telegram за номером.
Для цього потрібен серверний процес із легальним Telegram API-доступом, який має право бачити контакт або приймає дані від користувача через бота. У таблиці вже є поля `telegram_username`, `telegram_photo_url` / `master_photo_url`, куди сервер зможе записувати готовий URL.
